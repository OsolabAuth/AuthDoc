# AuthPortal Production Hardening

## 目的

AuthPortal の画面から開発用の初期値や検証コード表示を外し、本番向けに誤操作しにくい状態へ整理する。

対象は既存の認証・MFA・強化認可系画面であり、API仕様は変更しない。

## 対象画面

- `/login`
- `/mfa`
- `/password/change`
- `/password/reset`
- `/account/withdrawal`

## 方針

### 開発用初期値を削除する

以下のような開発確認用の値を画面初期値に入れない。

- `demo@example.com`
- `Passw0rd!`

ログインメールアドレス、パスワード、強化認可トークンは利用者が明示的に入力する。

### MFAコードを画面に表示しない

メールMFA開始APIのレスポンスに検証コードが含まれる場合でも、AuthPortal 画面には表示しない。

理由:

- 本番画面で認証コードを露出させないため
- 操作ログや画面共有にコードが残るリスクを下げるため
- メールで受け取ったコードを入力する実際の利用フローに合わせるため

### 必須入力をHTML側でも表現する

API側の検証を正とするが、画面側でも最低限の `required` を付け、空送信を減らす。

対象:

- メールアドレス
- パスワード
- 新しいパスワード
- 生年月日
- MFAコード
- 強化認可トークン

## 画面別変更

### ログイン画面

- メールアドレス初期値を空にする
- パスワード初期値を空にする
- メールアドレスとパスワードを必須入力にする

### MFA画面

- メールアドレス初期値を空にする
- メールMFA開始後に開発用コードを表示しない
- メールMFA開始後にコード入力欄を自動入力しない
- メールアドレス、メールコード、Authenticatorコードを必須入力にする

### パスワード変更画面

- メールアドレス初期値を空にする
- 現在パスワード、新パスワード、強化認可トークンを必須入力にする

### パスワードリセット画面

- メールアドレス、生年月日、新パスワードを必須入力にする

### 退会画面

- メールアドレス初期値を空にする
- パスワード、強化認可トークンを必須入力にする

## テスト観点

- `npm run typecheck` が成功する
- `npm run build` が成功する
- 画面コードに `demo@example.com` と `Passw0rd!` が残っていない
- MFA画面に開発用コード表示が残っていない
- 対象フォームの主要入力に `required` が付与されている
