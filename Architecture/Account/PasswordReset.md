# パスワードリセット設計

## 目的

ログインできないユーザーが、ログインメールアドレス、生年月日、メール認証コードを使ってパスワードを再設定できるようにする。

通常ログインセッションや既存アクセストークンは要求しない。ただし、メールアドレスだけでリセット用コードを送信しない。登録済みの生年月日と一致した場合だけメールコードを送信する。

## 基本方針

- ログインメールアドレスを識別子として使う。
- 生年月日はパスワードリセット開始時と完了時の両方で検証する。
- メールコードは6桁、有効期限付き、一度使ったら消費する。
- 開始APIはアカウント有無や生年月日一致をレスポンスで露出しない。
- 完了APIはメールコード、生年月日、新パスワードをすべて検証してからパスワードを更新する。
- 新パスワードは通常のパスワードポリシーを満たす必要がある。

## フロー

1. ユーザーがメールアドレスと生年月日を入力する。
2. Portal が `POST /password/reset/start` を呼び出す。
3. AuthFoundation は入力形式を検証する。
4. AuthFoundation は登録済みメールアドレスと生年月日が一致する場合だけメールコードを送信する。
5. レスポンスは常に `reset_challenge_started` とし、アカウント状態を露出しない。
6. ユーザーがメールコードと新しいパスワードを入力する。
7. Portal が `POST /password/reset` を呼び出す。
8. AuthFoundation はメールコード、生年月日、新パスワードを検証する。
9. 条件を満たす場合だけパスワードハッシュを更新する。

## セキュリティ要件

- `POST /password/reset/start` はメールアドレスと生年月日が一致しない場合でも成功レスポンスを返す。
- メールコードはレスポンスに含めない。
- メールコード検証を生年月日検証より先に実行し、メールコードがない状態で生年月日を探索しにくくする。
- 生年月日不一致、メールコード不一致、存在しないユーザーは認可失敗として扱う。
- パスワード更新後、既存のログインセッションやリフレッシュトークンの扱いは別途トークン失効設計に従う。

## エラー方針

| ケース | レスポンス |
|---|---|
| メール形式不正 | `400 invalid_request` |
| 生年月日形式不正 | `400 invalid_request` |
| メールコード未入力 | `400 invalid_request` |
| 新パスワードポリシー不一致 | `400 invalid_request` |
| メールコード不一致 | `401 invalid_token` |
| 生年月日不一致 | `401 invalid_token` |

## 初期実装範囲

- `POST /password/reset/start`
- `POST /password/reset`
- Portal の `/password/reset` 画面
- API Tester シナリオ
- 単体テスト
